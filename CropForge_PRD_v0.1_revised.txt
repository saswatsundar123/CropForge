1. Executive Summary

CropForge is an open-source, code-first virtual farm runtime that enables
agricultural researchers to define, execute, and visually analyse crop simulations
entirely from Python code. It is distributed as a pip-installable package and runs
completely on the researcher's local machine — no internet connection, no account,
no institutional server required.

Unlike DSSAT or APSIM, CropForge does not prescribe the model — the researcher's
equations are the model. CropForge provides the infrastructure: the time-stepping
engine, spatial field representation, environment resolver, state logger, and visual
frontend. Researchers use this infrastructure to express and run their own
process-based models. The accuracy of any simulation is a function of the model the
researcher writes, not of CropForge itself. CropForge guarantees faithful execution
of that model and precise, reproducible visual output.

Reference implementations for common growth processes are provided as starting
points in cropforge.examples (available from v0.2.0). These are clearly labelled as
illustrative material, not validated models, and are intended as copy-paste starting
blocks for researchers new to the platform.

Core Thesis: CropForge is to crop modelling what Python is to computing: a runtime
that makes no claim about what you build with it, but ensures that what you build
runs correctly, reproducibly, and visibly.

  ---------------------|-----------------------------------------------

  Distribution         | PyPI (pip install cropforge)

2. Problem Statement

2.1 The Gap in Current Tools

The agricultural research community currently has access to two categories of
simulation tools, and neither fully serves the needs of researchers who work with
novel crops, novel models, or novel experimental designs.

  Dimension            | Legacy Models (DSSAT / APSIM) | CropForge
  ---------------------|-------------------------------|---------------------------
  Crop coverage        | 22-30 hardcoded crops         | Unlimited - researcher-defined
  Model authorship     | Fixed by the development team | Researcher writes the model
  Input data format    | Proprietary, rigid schemas    | CSV, JSON, NetCDF, shapefile
  Visualisation        | Tables and basic 2D charts    | 3D field view + rich dashboard
  Reproducibility      | Parameter files, hard to share| A single Python script
  Extensibility        | Only through core team        | pip packages by the community
  Distribution        | GUI installer, institution-specific | pip install in terminal

2.2 The Reproducibility Crisis in Crop Modelling

A significant and underappreciated problem in published crop modelling research is
the inability to reproduce simulation results. Parameter files are lost, software
versions change, institutional licences expire. CropForge addresses this
structurally: because a simulation is a Python script, it can be shared as a file,
committed to a repository, attached to a journal submission, and re-run by any
reviewer who has Python installed.

2.3 What CropForge Does Not Try to Solve

Important Scope Boundary: CropForge does not claim accuracy for any simulation.
The researcher's model code is responsible for accuracy. CropForge faithfully
executes that code and visualises the output.

CropForge is not intended as a replacement for field trials. It is intended to
narrow the hypothesis space before field trials, reducing the resource expenditure
required to arrive at meaningful experimental designs. It is also not a breeding
platform in the genetic sense — it does not perform genomic selection, BLUP
calculations, or marker-trait association analysis.

4. System Architecture

4.1 Architectural Overview

CropForge is composed of four distinct layers. These layers are strictly separated:
the runtime engine has no knowledge of the visualisation layer, and the
visualisation layer has no knowledge of the model logic. They communicate only
through the state log file.

  Layer                     | Responsibility
  --------------------------|---------------------------------------------------
  Layer 1: Farm Definition  | The Python classes and decorators that the
  API                       | researcher uses to define their farm, crops,
                            | soil, weather, and model logic.
  Layer 2: Runtime Engine   | The time-stepping engine, spatial grid manager,
                            | event system, environment resolver, and state
                            | logger. Executes the researcher's model.
  Layer 3: State Log        | A structured Parquet file written by the engine,
                            | containing the full state of the simulation at
                            | every timestep. This is the handoff point between
                            | compute and visualisation.
  Layer 4: Visual Frontend  | A local web application (served at localhost)
                            | that reads the state log and renders the 3D farm
                            | view, metrics dashboard, event log, and farm
                            | inspector.

4.2 The Compute-First Principle

The simulation runs entirely to completion before the visual frontend is launched.
This is a deliberate architectural decision. The consequences are:

  - The visual playback is smooth regardless of model complexity, because it is
    replaying a log, not computing in real time.
  - The user can scrub to any point in time instantly.
  - The computational load is decoupled from the rendering load.
  - Multiple visualisation sessions can be opened from the same run without
    re-executing the model.

This principle also means that all frontend data-serving is read-only against an
already-complete Parquet file. There is no live bidirectional data stream between
the engine and the frontend.

5. The SimulationState Schema

The SimulationState schema is the most critical design decision in the entire
project. Everything — the runtime engine, the state logger, the visualisation
frontend, and the user-facing API — reads from or writes to this schema. It must
be stable before any other code is written.

The on-disk representation of SimulationState (the Parquet schema) is specified
in full in Section 16.

Design Rule: Every object in SimulationState carries a 'custom' dictionary. This
allows researchers to attach arbitrary variables without modifying the core schema.
CropForge will log all custom fields automatically.

5.1 PlantState

  @dataclass
  class PlantState:
      plant_id: str                      # Unique identifier, e.g. "r04c07"
      row: int                           # Row index in field grid
      col: int                           # Column index in field grid
      age_days: int = 0                  # Days since sowing
      lai: float = 0.0                   # Leaf Area Index (m2/m2)
      biomass_g: float = 0.0             # Total above-ground dry biomass (g)
      height_cm: float = 0.0             # Plant height (cm)
      root_depth_cm: float = 0.0         # Current rooting depth (cm)
      stress_index: float = 0.0          # Composite stress (0=none, 1=dead)
      alive: bool = True
      phenological_stage: str = "germination"  # e.g. "vegetative", "flowering"
      custom: Dict[str, Any] = field(default_factory=dict)

5.2 SoilVoxelState

  @dataclass
  class SoilVoxelState:
      row: int
      col: int
      layer: int                         # Soil depth layer index (0 = topsoil)
      depth_top_cm: float                # Top of this layer (cm)
      depth_bottom_cm: float             # Bottom of this layer (cm)
      moisture_pct: float                # Volumetric water content (%)
      nitrogen_kg_ha: float              # Available nitrogen (kg/ha)
      bulk_density: float                # g/cm3
      penetration_resistance: float      # MPa — enables hard pan modelling
      custom: Dict[str, Any] = field(default_factory=dict)

5.3 FieldState

  @dataclass
  class FieldState:
      day: int
      plants: List[PlantState]
      soil: List[List[List[SoilVoxelState]]]  # [row][col][layer]
      elevation_grid: np.ndarray              # 2D array of relative elevation (m)
      events_fired: List[str]                 # Names of events that fired this day
      custom: Dict[str, Any] = field(default_factory=dict)

5.4 EnvironmentState

  @dataclass
  class EnvironmentState:
      day: int
      doy: int                           # Day of year
      temp_max_c: float
      temp_min_c: float
      temp_mean_c: float
      radiation_mj_m2: float             # Incoming solar radiation
      rainfall_mm: float
      et0_mm: float                      # Reference evapotranspiration
      wind_speed_ms: float               # Wind speed (m/s — SI units internally)
      humidity_pct: float
      co2_ppm: float = 415.0
      custom: Dict[str, Any] = field(default_factory=dict)

6. Farm Definition API

This is the interface the researcher interacts with. It must be clean, minimal,
and read like agricultural domain language, not software engineering language.

6.1 Basic Usage Example

  from cropforge import Farm, Field, Crop, Weather, Soil, Event

  # Define the farm
  farm = Farm(name="Trial 2026-A", location=(23.4, 85.3))

  # Define fields
  field_a = Field(
      name="Plot A",
      rows=40, cols=60,
      area_ha=2.5,
      elevation_profile="slope_1pct_N"  # or np.ndarray DEM
  )

  field_b = Field(
      name="Plot B",
      rows=40, cols=60,
      area_ha=2.5
  )

  # Attach inputs
  field_a.set_soil(Soil.from_csv("soil_a.csv", layers=5))
  field_a.set_weather(Weather.from_csv("weather_2026.csv"))
  field_a.set_crop(Crop(species="maize", variety="custom", sowing_doy=120))

  field_b.set_soil(Soil.from_csv("soil_b.csv", layers=5))
  field_b.set_weather(Weather.from_csv("weather_2026.csv"))
  field_b.set_crop(Crop(species="maize", variety="custom", sowing_doy=120))

  farm.add_field(field_a)
  farm.add_field(field_b)

  # Register management events
  farm.add_event(Event.irrigation(field="Plot A", interval_days=15, amount_mm=30))
  farm.add_event(Event.irrigation(field="Plot B", interval_days=15, amount_mm=30))

6.2 The @farm.step Decorator

Step functions define all model logic. The phase argument is an integer that
controls the execution order among all registered step functions within a single
simulation run. Lower phase numbers execute first. This enforces deterministic
ordering when multiple step functions interact through shared state (e.g., a soil
evaporation step that must complete before a transpiration step can read the
updated soil moisture).

  # Soil processes run first (phase=1), plant physiology runs second (phase=2).
  # This ensures transpiration reads soil moisture after evaporation has reduced it.

  @farm.step(interval="daily", phase=1)
  def soil_evaporation(state, env):
      for row in state.soil:
          for col_voxels in row:
              topsoil = col_voxels[0]
              topsoil.moisture_pct -= compute_evaporation(env.et0_mm, env.wind_speed_ms)
              topsoil.moisture_pct = max(0.0, topsoil.moisture_pct)
      return state

  @farm.step(interval="daily", phase=2)
  def growth_model(state, env):
      for plant in state.plants:
          if not plant.alive:
              continue

          # Researcher's own equations
          delta_lai = compute_lai_increment(
              env.temp_mean_c, env.radiation_mj_m2, plant.age_days
          )
          plant.lai = max(0, plant.lai + delta_lai)
          plant.biomass_g += compute_biomass(plant.lai, env.co2_ppm, env.et0_mm)
          plant.height_cm += compute_height(plant.biomass_g, plant.age_days)

          # Check permanent wilting point using updated soil state
          soil_layer = state.soil[plant.row][plant.col][0]
          if soil_layer.moisture_pct < PWP_THRESHOLD:
              plant.stress_index += 0.12
          if plant.stress_index >= 1.0:
              plant.alive = False

      return state

  # Run the simulation
  farm.run(days=120)

  # Launch the visual frontend
  farm.visualize()

Phase rules:
  - phase must be a positive integer. Default is phase=0 if the argument is omitted.
  - If two or more step functions share the same phase integer, their relative
    execution order within that phase is undefined. The engine will emit a warning
    to the Event Log at the start of the run if this condition is detected, alerting
    the researcher that ordering within that phase is non-deterministic.
  - Step functions with no phase argument default to phase=0. If multiple unphased
    steps are registered, the same warning is emitted.
  - Phase integers do not need to be contiguous. phase=1 and phase=10 are valid;
    phase=1 will run first.

6.3 Event System

Events are discrete actions that fire on specific days or at intervals. The
researcher registers them before running. Built-in event types in v0.1:

  - Event.irrigation(field, interval_days, amount_mm)
      Adds moisture to all soil cells on schedule.
  - Event.fertiliser(field, day, n_kg_ha)
      Adds nitrogen to topsoil layer.
  - Event.custom(field, day, fn)
      Calls an arbitrary function on a specific day.

6.4 Error Handling Contract

If the researcher's step function raises an unhandled exception during a simulation
run, the following behaviour is guaranteed:

  1. The run halts immediately. No further timesteps are executed.
  2. The Parquet state log is flushed and saved with all timesteps completed up to
     and including the last successful day. The partial log is valid and can be
     opened with farm.visualize() to inspect the simulation up to the point of
     failure.
  3. A crash log is written to the session output directory as
     cropforge_crash.log. The crash log contains: the day number on which the
     exception occurred, the name of the step function that raised it, and the
     full Python traceback.
  4. A CropForgeStepError is raised in the terminal with a summary message
     pointing the researcher to the crash log.

Silent failure is explicitly not permitted. CropForge will never skip a failed
timestep and continue silently. If a step function raises, the run stops.

Custom logic within step functions may catch their own exceptions and handle them
internally. Only uncaught exceptions propagate to the engine's halt behaviour.

6.5 farm.visualize() Pre-flight Check

When the researcher calls farm.visualize(), the engine performs a pre-flight check
before launching the FastAPI server:

  1. It checks for a valid Parquet state log from the most recent farm.run() call
     in the current session, or from the path passed via farm.visualize(log=path).
  2. If no log file is found, or the log file is empty or corrupt, CropForge raises
     a CropForgeVisualizeError with the message:
     "No valid simulation log found. Run farm.run() before calling
     farm.visualize(), or pass an explicit log path via farm.visualize(log=path)."
  3. If a log file exists but was produced by a different version of CropForge,
     a version mismatch warning is printed to the terminal. The visualisation
     proceeds but the researcher is advised to re-run the simulation.

7. Frontend Specification

7.1 Launch Behaviour

When the researcher calls farm.visualize(), CropForge performs the pre-flight
check (Section 6.5), then starts a local FastAPI server, opens the researcher's
default browser to http://localhost:7860, and serves the frontend from static
files. No internet connection is required. The server shuts down when the browser
tab is closed or the terminal process is interrupted.

7.2 Panel Layout

  Panel                              | Description
  -----------------------------------|----------------------------------------
  Panel 1: 3D Farm View (60%)        | Three.js rendered field. Plants as
                                     | instanced meshes, height-mapped to
                                     | biomass, colour-mapped to any variable
                                     | the researcher selects. Timeline scrubber
                                     | at bottom controls playback. Click any
                                     | plant to inspect it.
  Panel 2: Metrics Dashboard         | Plotly time-series charts auto-generated
  (upper right, 40%)                 | from all logged state variables. Supports
                                     | multi-field overlay and scenario
                                     | comparison. Exportable to PNG and CSV.
  Panel 3: Event Log                 | Scrollable log showing which events fired
  (lower right, partial)             | on each day, warnings (e.g. PWP
                                     | violation, unphased step conflict), and
                                     | any errors raised in the researcher's
                                     | step function.
  Panel 4: Farm Inspector            | Opens when a plant is clicked. Shows
  (sidebar, collapsed by default)    | full PlantState and SoilVoxelState history
                                     | for that cell. Includes vertical soil
                                     | cross-section chart showing moisture and
                                     | nitrogen by layer.

7.3 3D View Specification

  Feature              | Specification
  ---------------------|------------------------------------------------------
  Renderer             | Three.js r152 (browser-based WebGL, no GPU driver
                       | dependency)
  Plant representation | Instanced cylinder meshes — height = plant.height_cm
  v0.1                 | (scaled), radius = LAI proxy
  Colour mapping       | User selects variable from dropdown; full HSL gradient
                       | applied across field
  Dead plants          | Colour shifts to #8B6914 (dried brown); height
                       | collapses to ground level over 3 days
  Soil surface         | 2D texture map of topsoil moisture, visible between
                       | plant instances
  Timeline scrubber    | Draggable slider across simulation days;
                       | play/pause/2x/5x speed controls
  Click to inspect     | Raycasting — clicking a mesh highlights it and opens
                       | Panel 4
  Multi-field view     | Side-by-side 3D fields for farms with multiple plots

Data serving for the 3D view:

The FastAPI server does not serve JSON object trees to the Three.js frontend.
Instead, per-timestep field state is served as flat binary Float32Array buffers
containing plant positions, heights, and colour-mapped variable values, packed in
the order that Three.js InstancedMesh attribute arrays expect. This eliminates
JSON serialisation and deserialisation overhead on the main thread.

On page load, the frontend preloads all timestep buffers for the entire simulation
into browser memory before enabling the timeline scrubber. For a standard 120-day
simulation, this is a one-time upfront cost (typically under 2 seconds on a modern
laptop) after which all scrubbing is instantaneous with zero additional network
requests. The preload progress is shown as a loading bar before the 3D view
activates.

The Parquet-to-binary conversion is performed once per session by the FastAPI
server at startup and held in memory. The server does not re-read Parquet on each
scrub event.

7.4 What the 3D View Does NOT Do in v0.1

Scope Boundary: Procedural leaf geometry deformation (wilting animation),
individual leaf/stem rendering, and voxel soil cross-section 3D rendering are
v1.0 features. v0.1 uses simplified mesh representations that convey the same
scientific information.

8. Data Input Specification

8.1 Weather Data (CSV)

Expected columns (column names are flexible, mapped via keyword arguments):

  date, tmax_c, tmin_c, radiation_mj, rainfall_mm, humidity_pct, wind_ms

  # Usage
  Weather.from_csv("weather.csv",
      date_col="date",
      tmax_col="tmax_c",
      radiation_col="radiation_mj"
  )

Multi-field weather resolution:

Each field in a farm resolves its weather independently. Two fields may be given
different weather CSV files, allowing the researcher to test spatially varied
weather inputs within a single farm run. The environment resolver (Layer 2) reads
from the weather source attached to each field object, not from a farm-level
weather source.

If two fields are given the same weather file (as in the usage example in Section
6.1), the file is loaded once and the parsed weather object is shared in memory —
it is not read twice from disk.

There is no interpolation or spatial weather modelling between fields in v0.1.
Each field's weather is treated as uniform across its spatial extent.
Spatially explicit weather (e.g., microclimate variation within a single field)
is out of scope until the hydrology layer in v0.3.0.

8.2 Soil Profile (CSV)

One row per soil layer per field cell, or a single profile applied uniformly
across the field:

  layer, depth_top_cm, depth_bottom_cm, moisture_pct, n_kg_ha,
  bulk_density, pen_resistance_mpa

  # Usage
  Soil.from_csv("soil.csv", apply="uniform")          # same profile across all cells
  Soil.from_csv("soil_spatial.csv", apply="spatial", rows=40, cols=60)

8.3 Elevation / Topology (NumPy array or CSV)

  import numpy as np
  # Manual DEM
  dem = np.zeros((40, 60))
  for row in range(40):
      dem[row, :] = row * 0.005  # 0.5% slope northward
  field.set_elevation(dem)

  # Or from file
  field.set_elevation(Field.elevation_from_csv("dem.csv"))

9.1 cropforge.examples (v0.2.0)

The cropforge.examples module is a curated set of reference implementations of
simple process-based growth models. It is shipped as part of the cropforge package
from v0.2.0 onwards.

Important classification: cropforge.examples models are illustrative only. They
are not validated against field data and make no claim of agronomic accuracy. They
exist to give researchers a concrete, runnable starting point they can read,
modify, and build on.

Planned examples at v0.2.0 launch:

  - examples.wheat_rue
      A minimal RUE (Radiation Use Efficiency) based wheat growth model.
      Models biomass accumulation from intercepted radiation, simple phenology
      (vegetative, heading, grain fill), and water stress using soil moisture
      relative to PWP. Approximately 80 lines of model code.

  - examples.maize_water_stress
      A simple maize model with water stress multiplier. Demonstrates multi-layer
      soil water balance interaction with transpiration and illustrates the use of
      the phase= argument to enforce soil-before-physiology execution order.

Each example is a complete, runnable Python script with inline comments explaining
every equation. They are explicitly not in the cropforge core runtime — they are
in a separate submodule so researchers can inspect, fork, and diverge from them
without touching core infrastructure.

10. Reference Example: wheat_basic.py

wheat_basic.py is the primary onboarding artifact for CropForge. It is shipped in
examples/ at v0.1.0 and is the deliverable for Phase 1 of the build. It must be a
complete, runnable simulation that a new user can execute immediately after
pip install cropforge without any additional setup beyond providing the two
specified data files.

10.1 Simulation Scenario

  Crop           : Wheat (generic variety, no cultivar-specific parameters)
  Field size     : 20 rows x 30 cols (600 plant positions)
  Simulation     : 90 days (approximately tillering to grain fill for wheat)
  Soil           : 3 layers (0-20cm, 20-40cm, 40-60cm), uniform profile
  Weather        : 90-day synthetic CSV included in the examples/ directory
                   (generated from a mid-season wheat climate, not from real
                   station data, clearly labelled synthetic)
  Management     : One irrigation event at day 30 (30mm), one fertiliser
                   event at day 15 (40 kg N/ha)

10.2 Model Logic Included

wheat_basic.py implements the following equations, all clearly commented:

  Biomass accumulation:
      delta_biomass = RUE * APAR * (1 - water_stress_factor)
      where APAR = radiation_mj_m2 * (1 - exp(-k * lai))
      RUE = 1.2 g/MJ (a commonly cited illustrative value)
      k = 0.5 (light extinction coefficient)

  LAI increment:
      delta_lai = f(temp_mean_c, age_days)
      Simple linear thermal time accumulation above a base temperature of 0C

  Water stress factor:
      stress = max(0, min(1, (moisture_pct - PWP) / (FC - PWP)))
      where FC = field capacity (set in soil CSV), PWP = permanent wilting point

  Height:
      height_cm = k_h * biomass_g^0.5 (allometric proxy)

  Plant death:
      if stress_index >= 1.0: plant.alive = False

10.3 Data Files Included

  examples/data/wheat_synthetic_weather_90d.csv
      90 rows. Columns: date, tmax_c, tmin_c, radiation_mj, rainfall_mm,
      humidity_pct, wind_ms. Values represent a temperate wheat growing season.
      File header includes a comment: "SYNTHETIC DATA — NOT FROM A WEATHER STATION"

  examples/data/wheat_uniform_soil_3layer.csv
      3 rows (one per layer). Columns: layer, depth_top_cm, depth_bottom_cm,
      moisture_pct, n_kg_ha, bulk_density, pen_resistance_mpa.
      Field capacity and PWP are embedded as comments in the file header.

10.4 Expected Output

After farm.run(days=90):
  - A valid Parquet log is written to ./cropforge_output/wheat_basic_<timestamp>/
  - The log contains 90 timesteps, 600 plant records per timestep, 3 soil layers
    per plant position
  - farm.visualize() opens http://localhost:7860 showing 600 plant cylinders
    growing over 90 days with colour-mapped biomass

Success criterion: the full run completes in under 10 seconds on a laptop with
4 CPU cores and 8 GB RAM. The Parquet log is under 50 MB for this field size
(see Section 11 for storage estimates).

11. Technical Constraints and Decisions

  Decision                          | Rationale
  ----------------------------------|-------------------------------------------
  Python 3.10+ only                 | Required for dataclasses with field()
                                    | defaults and match-case syntax used in
                                    | the event system.
  Parquet for state log             | Columnar format reads 50-100x faster than
  (not HDF5 or CSV)                 | CSV for the time-slice queries the
                                    | frontend makes. Smaller file size than
                                    | HDF5 for structured data.
  FastAPI for local server          | Async support allows the dashboard and 3D
  (not Flask)                       | view to share one server process with no
                                    | threading issues.
  Three.js for 3D                   | Runs in browser with no additional
  (not Unity, Godot, or Blender)    | installation. WebGL is available on every
                                    | modern device. Dependency is a CDN import.
  Dash for metrics dashboard        | Dash integrates with Plotly natively and
  (not Streamlit)                   | supports custom layout. Streamlit's
                                    | reruns-on-interaction model conflicts with
                                    | the playback architecture.
  NumPy for spatial grid            | The soil grid is a 3D array operation.
  (not Pandas)                      | NumPy vectorised operations are 10-100x
                                    | faster than Pandas for matrix math.
  Wind input in m/s internally      | All meteorological inputs are standardised
  (not knots)                       | to SI units inside the environment
                                    | resolver. The loader accepts any unit with
                                    | a conversion argument.
  Binary Float32Array for 3D        | Eliminates JSON serialisation on the main
  frontend data (not JSON)          | thread. Three.js InstancedMesh attributes
                                    | consume typed arrays directly. All timestep
                                    | buffers preloaded at page load for
                                    | instantaneous scrubbing.

11.1 Storage and Memory Footprint Estimates

Researchers should be aware of approximate storage costs before running large
simulations.

Parquet state log size (estimated, with default Snappy compression):

  Field size    | Days | Soil layers | Estimated log size
  --------------|------|-------------|--------------------
  20 x 30       |  90  |      3      | ~15-25 MB
  (wheat_basic) |      |             |
  40 x 60       | 120  |      5      | ~150-250 MB
  (maize, 2400  |      |             |
  plants)       |      |             |
  100 x 100     | 200  |      5      | ~1.5-2.5 GB
  (large field) |      |             |

These estimates assume core state variables only. Each custom field added to
PlantState.custom adds proportionally to the log size.

Runtime memory during simulation (Python process):

  - The engine holds only the current timestep's SimulationState in memory during
    the run. It does not accumulate all timesteps in RAM. Memory usage during
    compute is therefore O(field_size * soil_layers), not O(days * field_size).
  - For a 40x60 field with 5 soil layers: approximately 150-300 MB RAM during
    the simulation run.
  - For a 100x100 field with 5 soil layers: approximately 500 MB - 1 GB RAM.

Frontend memory (browser, after preload of all timestep buffers):

  - Float32Array per plant per timestep: approximately 7 floats
    (x, y, z position, height, colour R, G, B) = 28 bytes per plant per day.
  - For a 40x60 field (2400 plants) at 120 days:
    2400 * 120 * 28 bytes = ~8 MB in browser memory. This is negligible.
  - For a 100x100 field (10000 plants) at 200 days:
    10000 * 200 * 28 bytes = ~56 MB in browser memory. Well within normal limits.

Disk space recommendation: Researchers running large fields (>5000 plants) for
long seasons (>150 days) should ensure at least 5 GB of free disk space before
running simulations. CropForge will check available disk space before starting a
run and emit a warning if space is insufficient for the estimated log size.

12. Competitive Positioning

12.1 Relationship to DSSAT and APSIM

CropForge is not a competitor to DSSAT or APSIM. It is a complementary
infrastructure layer. The intended positioning is:

  - Researchers who use DSSAT can import DSSAT outputs as a data source and
    visualise them in CropForge. (v0.4 feature)
  - Researchers building novel crop models that DSSAT does not support use
    CropForge as their simulation runtime from the start.
  - CropForge's crop coverage is unlimited by design — any researcher who
    publishes a cropforge-{crop} package on PyPI extends the ecosystem.

12.2 The Trust Question

Scientists will ask: how do I trust CropForge's results? The answer is that
CropForge makes no claims about results. It makes claims only about execution
fidelity — that what you code is what runs, and what ran is what you see. The
scientific validity of the output is the researcher's responsibility and is
established through the same peer review process they already use.

Citation Model: Published papers using CropForge will cite:
"Simulation conducted using CropForge v0.2.1 (Saswat Sundar Rath, 2026). Code available
at [DOI]." The version number and attached code together constitute a fully
reproducible experimental record.

12.3 Why the Breeding Claim Was Removed

Earlier discussions positioned CropForge as a tool for the plant breeding
industry broadly. This positioning has been removed based on expert review.
Breeding as a discipline requires statistical genetics infrastructure (BLUP, GEBV,
marker-trait association) that CropForge does not provide and is not designed to
provide. The valid use case for breeding is narrow and specific: ideotype design
(simulating the effect of modified physiological traits on yield) and G×E scenario
testing. These will be documented as specific use cases, not a primary positioning.

16. Parquet State Log Schema

The Parquet state log is the contract between the runtime engine and the visual
frontend. Both sides must agree on this schema before either is implemented. The
schema is flat (denormalised) — nested Python objects are unpacked into columns.
This enables efficient columnar reads for time-slice queries.

16.1 Schema Overview

The log is a single Parquet file. Each row represents one plant at one timestep.
Soil state is joined at query time using (day, row, col, layer) as a compound key.

Plant table (one row per plant per day):

  Column name          | Parquet type   | Description
  ---------------------|----------------|------------------------------------------
  day                  | INT32          | Simulation day (1-indexed)
  field_name           | STRING         | Field identifier (e.g. "Plot A")
  plant_id             | STRING         | Plant identifier (e.g. "r04c07")
  row                  | INT16          | Row index in field grid
  col                  | INT16          | Column index in field grid
  age_days             | INT16          | Days since sowing
  lai                  | FLOAT          | Leaf Area Index (m2/m2)
  biomass_g            | FLOAT          | Above-ground dry biomass (g)
  height_cm            | FLOAT          | Plant height (cm)
  root_depth_cm        | FLOAT          | Rooting depth (cm)
  stress_index         | FLOAT          | Composite stress (0.0-1.0)
  alive                | BOOLEAN        | Whether plant is alive
  phenological_stage   | STRING         | e.g. "vegetative", "flowering"
  custom_json          | STRING         | JSON-serialised PlantState.custom dict

Soil table (one row per voxel per day):

  Column name          | Parquet type   | Description
  ---------------------|----------------|------------------------------------------
  day                  | INT32          | Simulation day
  field_name           | STRING         | Field identifier
  row                  | INT16          | Row index
  col                  | INT16          | Column index
  layer                | INT8           | Soil depth layer (0 = topsoil)
  depth_top_cm         | FLOAT          | Top of layer (cm)
  depth_bottom_cm      | FLOAT          | Bottom of layer (cm)
  moisture_pct         | FLOAT          | Volumetric water content (%)
  nitrogen_kg_ha       | FLOAT          | Available nitrogen (kg/ha)
  bulk_density         | FLOAT          | Bulk density (g/cm3)
  penetration_resistance | FLOAT        | Penetration resistance (MPa)
  custom_json          | STRING         | JSON-serialised SoilVoxelState.custom

Environment table (one row per day per field):

  Column name          | Parquet type   | Description
  ---------------------|----------------|------------------------------------------
  day                  | INT32          | Simulation day
  field_name           | STRING         | Field identifier
  doy                  | INT16          | Day of year
  temp_max_c           | FLOAT          |
  temp_min_c           | FLOAT          |
  temp_mean_c          | FLOAT          |
  radiation_mj_m2      | FLOAT          |
  rainfall_mm          | FLOAT          |
  et0_mm               | FLOAT          |
  wind_speed_ms        | FLOAT          |
  humidity_pct         | FLOAT          |
  co2_ppm              | FLOAT          |
  events_fired         | STRING         | JSON array of event names fired this day
  custom_json          | STRING         | JSON-serialised EnvironmentState.custom

16.2 Partitioning and Compression

  - The Parquet file is partitioned by field_name, then by day.
    This means the frontend can read all plants for a single day of a single field
    without scanning the full file.
  - Compression: Snappy (default pyarrow compression). Chosen for decompression
    speed over compression ratio, since the frontend makes many small random reads.
  - Row group size: one day's worth of plant records per row group. This aligns
    row group boundaries with the timeline scrubber's access pattern.

16.3 Custom Fields

  - PlantState.custom, SoilVoxelState.custom, and EnvironmentState.custom are
    serialised as JSON strings into the custom_json column of their respective
    tables.
  - This allows arbitrary researcher-defined variables to be logged without
    modifying the core schema.
  - The frontend reads custom_json and deserialises it for display in the Panel 4
    inspector and for making custom variables available in the colour-mapping
    dropdown in Panel 1.

16.4 Schema Stability Guarantee

  The Parquet schema defined in this section is frozen for all v0.x releases.
  New columns may be added in minor versions. No existing column will be renamed,
  retyped, or removed before v1.0. Each Parquet file records the CropForge version
  that produced it in the file-level metadata under the key "cropforge_version".
  This is used by the pre-flight check (Section 6.5) for version mismatch detection.

17. Appendix: Glossary

  Term               | Definition
  -------------------|-------------------------------------------------------------
  LAI                | Leaf Area Index — the total one-sided area of leaf tissue
                     | per unit ground surface area (m2/m2). A key indicator of
                     | canopy development.
  ET0                | Reference evapotranspiration — the rate of evapotranspiration
                     | from a standardised reference surface, used to estimate crop
                     | water demand.
  PWP                | Permanent Wilting Point — the soil moisture level below which
                     | plant roots can no longer extract water, causing irreversible
                     | wilting.
  Hard pan           | A dense, compacted soil layer with high penetration resistance
                     | that impedes root growth and water drainage.
  RUE                | Radiation Use Efficiency — the efficiency with which a crop
                     | converts intercepted solar radiation into dry biomass.
  Parquet            | A columnar binary file format optimised for read-heavy
                     | analytical workloads. Used by CropForge for state logging.
  FSP Model          | Functional-Structural Plant Model — a model that explicitly
                     | represents both the architecture and physiology of a plant.
  D8 algorithm       | A terrain analysis algorithm that routes water flow from each
                     | cell to one of its eight neighbours based on slope direction.
  Ideotype           | A theoretical plant type with a defined combination of traits
                     | considered ideal for a specific purpose or environment.
  SimulationState    | The complete data structure representing the state of the
                     | entire farm (all plants, all soil layers, all field metadata)
                     | at a single timestep.
  Phase (step)       | An integer argument to @farm.step that determines the
                     | execution order of step functions within a daily timestep.
                     | Lower phase numbers execute first.
  Float32Array       | A JavaScript typed array containing 32-bit floating point
                     | values. Used by CropForge's 3D frontend to consume instanced
                     | mesh attribute data directly without JSON deserialisation.
